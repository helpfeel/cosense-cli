# Cosenseのページを編集する手順書

CLI コマンドの仕様や実行形式は [SKILL.md](SKILL.md) を参照する。
各コマンドの正確な引数・入力形式・出力フォーマットは `cosense <command> --help` を見る。

## ワークフロー

以下の手順1〜7を順番に実行する。見出しに条件が付いている手順（2・3・7）は、条件に該当しない時は飛ばして次へ進む。

### 手順1: 編集対象のページを把握する

文面を考え始める前に、まず対象ページを読む。新規作成のつもりでも読む。ページが既に存在して、本文に前書きや書き込み方の指示が書かれている事がある。

このセッションでまだ `browsePage` していないページなら、先に `browsePage` で全貌を把握する。関連ページ一覧やInfoboxは `readPage` では得られず、文面とリンク選定の文脈になる。その後 `readPage` で編集に使うfieldを取得する。

注目するfield:

- `persistent`: `true` なら既存ページ編集、 `false` なら新規ページ作成
- `persistent: true` の時だけ top-level `id` を `previewEdit` の `<pageId>` に渡す
- 各 line の `id` は ops の anchor (insertBefore / replace / delete の対象) に使う
- `persistent: false` の時は `pageId` / `lineId` を使わず、 手順4で `previewEdit --new` を使う

編集指示のURLに `#<lineId>` fragmentが付いている時は、その行が書き込み位置の最優先anchorである。
本文を読んで別の場所が適切に見えても、fragmentが指す行を優先する。

### 手順2: 文面を組み立てる（新しい文を書く時・リンクを追加・変更する時）

新しい行・段落・ページを書き足す編集と、既存の文にリンク記法を追加・変更する編集で実行する。以下の編集ではこの手順を飛ばす。

- リンク記法に触れない、誤字修正・記法の手直し・行削除だけの編集
- ユーザーが書く文面を一字一句指定している編集

#### まわりのページのスタイルに合わせる

書き方のスタイルはproject毎に様々である。固定の文体規則を当てはめず、既存ページを観察して合わせる。

- シリーズもの（連番・日付入りタイトル等）に書く時は、前回のページを`browsePage`で読む
- 新規ページを作る時は、近いテーマの既存ページを1〜2枚`browsePage`で観察する
- 観察対象: 見出しの立て方、箇条書きの深さ、文体、アイコン記法の使われ方
- 自分（AI）の発言に署名する時は、project内で既に使われている自分を指すアイコン記法を探して合わせる

#### 参考にできるページが無い時

観察できる既存ページが無い時は、他の記法の文書構造を翻訳して持ち込まず、Cosenseの素の書き方をデフォルトにする。

- 見出しを立てる代わりに、文章の構造はインデントで、段落の区切りは空行で表現する
- `[* ]` 等の強調記法は強調であって見出し記法ではない。見出しとして使わない

#### 内部リンクを選定する

リンク記法`[タイトル]`はナレッジグラフへの接続である。文面上それっぽい語をリンク化するのではなく、project内に既にある知識ノードとの接続として選ぶ。ユーザーが張るリンクを明示している時（リンク不要の指定を含む）は、指定に従いこの選定を飛ばす。それ以外は以下を順に行う。

1. 文案からリンク候補を抽出する。ページの主題を構成する中心概念に絞る。文中の固有名詞の網羅ではない
2. 候補ごとに`searchVector`で既存ページを探す。表記ゆれ・類義語・英日訳は`searchFullText`のOR検索でまとめて拡張する。検索のコツは [read-page.md](read-page.md) の「ベクトル検索のコツ」と「良い起点が見つからない場合」を参照
3. 採用を迷う候補だけ`browsePage`または`list1hopLinks`で深掘りする。`linked`・`pageRank`・`descriptions`・関連ページ一覧の広がりに注目する。本文が無いページでも被リンクが多ければ、グラフ上のノードとして既に機能している
4. 候補ごとに採用・却下を「対象 — 理由」の1行形式で決める。既存ページに繋ぐ時は、リンク表記を既存ページタイトルに一致させる

リンク0本という結論も正当である。接続する価値のある既存ノードが無い文面に、無理にリンクを作らない。

#### リンク採用・却下の判断基準

採用するリンク:

- ページの主題を構成する概念で、既存ノード（被リンクを持つ空ページ含む）に繋がる語

却下するリンク:

- project内にまだ無い概念の語。新しいノードを立てる投機的な空リンクは人間が付けるものであり、AIは張らない
- 文面上それっぽいだけで、ページの主題を構成する概念ではない語（中心でない固有名詞・一般語）
- 主題と直接関係のない、被リンク数の突出したHUBページへのリンク
- 文字列は一致するが文脈が違うページ
- 同一ページへ既にリンク済みの語（重複リンク）

却下はリンク記法にしないだけであり、語そのものは平文で書いてよい。

#### コメントを追記する時

- 意思決定のリソースになる内容だけ書く。本文の要約や自明な補足は書かない
- ページ全体に満遍なく散らさない。読み手の判断を変えうる少数の点に絞る
- 既存の行への返信は、その行の子行としてインデントを1段深くして書く

### 手順3: リンク計画の承認を得る（採用するリンクがある時）

手順2でリンクの採用・却下を決めたら、`previewEdit` を呼ぶ前に採用・却下の一覧を理由付きで提示し、AskUserQuestion で以下の2択を提示する。

| 選択肢                      | 何が起こるか                                               | どんな時に向くか                             |
| --------------------------- | ---------------------------------------------------------- | -------------------------------------------- |
| **この計画で書く** （推奨） | 採用リンクを含む文面を確定し、手順4へ進む                  | 提示された接続で意図が表現できている         |
| **リンクを調整する**        | ユーザーの指示に従って採用・却下を入れ替え、計画を作り直す | 繋ぎたいページが別にある、リンクが過剰・不足 |

#### 質問をスキップする条件

- 手順2でリンク選定を行わなかった、または採用したリンクが1つも無い（全候補を却下した）場合は、質問せず手順4へ進む。却下理由は完了報告に書く
- 別のSkillや上位Agentから呼び出され、ユーザーと直接対話していない場合は、採用案をデフォルトとして進む

質問は1回の編集につき1回だけ。previewを作り直してもリンク計画が変わらないなら再質問しない。

### 手順4: previewEdit でdry-runする

- 既存ページ: `previewEdit <projectUrl> <pageId>` (stdin = ops JSON)
- 新規ページ: `previewEdit --new <projectUrl>` (stdin = プレーンテキスト本文。 1行目がページタイトル)

入力形式・出力フォーマットは `cosense previewEdit --help` を参照。

ops の組み立てwisdom:

- 特定行を複数行に分割したい時は、対象 lineId の直前に複数行を `insertBefore` してから対象行を `delete` する。 逆順にすると anchor 不在で失敗する
- ページ末尾に追記したい時は、 anchor に `_end` を指定する。ただし末尾に空行が連なっているページでは、空行群の先頭に`insertBefore`する方がよい。`_end`だと追記が余白の後ろにぶら下がる

#### 記法衝突の点検

preview出力はplain textであり、記法の描画結果は確認できない。previewEditに渡す前にtextを点検する。

- `#数字`はhashtag記法として解釈される。PR番号等はバッククォートで囲む
- 正規表現や配列など`[]`を含むコード断片はリンク記法と衝突する。バッククォートで囲む
- `[URL]`とブラケットで囲まれた画像URLは埋め込み表示になる。埋め込む意図がある時だけ囲む

#### Windowsで書き込む時

Windows 環境では本文や ops JSON を UTF-8 ファイルに書き、`previewEdit --input-file <path>` で渡す。
PowerShell の pipe 経由だと日本語が `?` に化けるため。

### 手順5: 出力を読んで適用後の page を確認する

意図通りの変更になっているかを確認する。意図と違えば ops を組み直して手順4からやり直す。

リンク計画と突合する: 採用したリンクだけが入っているか、却下した語がリンク記法になっていないか。
関連ページ一覧はpreview出力に含まれないので、リンク接続の確認は手順7で行う。

### 手順6: submitEdit で確定する

`submitEdit <projectUrl> <previewId>` を実行する。 `commitId:` が返ったら書き込み成功。 成功後は同じ `previewId` も同じ編集意図も再送しない。

`submitEdit` の response を受け取れなかった、 または結果が曖昧な時は、 再submitする前に `readPage` で対象ページを読み、 意図した変更が既に反映されていないか確認する。

### 手順7: リンク接続を確認する（リンクを追加・変更した時）

リンクを追加・変更した編集の時だけ実行する。それ以外の編集は手順6で終了する。

submit成功後、対象ページを`browsePage`で1回だけ読み、出力末尾の関連ページ一覧に書き込んだリンクの接続先が現れているか確認する。

一覧に現れない正常なケースが2つある。本文の無い空ページへのリンクは、反映を待っても一覧に現れない。本文のあるページへのリンクも、書き込み直後は反映が遅れて現れない事がある。現れない時は、書いたリンク表記が検索で得たページタイトルと一致しているかだけ確かめる。一致していれば接続は成立しているので何もしない。違っていたら表記を直す編集を行う。

- 関連ページの深掘りはしない。接続の確認だけで終了する

## 失敗時の復旧

HTTPエラーの正確な意味は `cosense previewEdit --help` / `cosense submitEdit --help` を確認する。
ここでは編集ワークフロー上の復旧方針だけを書く。

### previewEdit

- HTTP 404: pageId が現在のページを指していない。 `readPage` からやり直し、最新の pageId を取得する。対象ページが存在しない状態なら新規作成の手順に切り替える
- HTTP 422: ops が現在のページ状態に対して不正。 `previewEdit --help` と現在の `readPage` 出力を見直して ops を組み直す

### submitEdit

- HTTP 400: preview生成時と違う projectUrl で submit している。preview生成時と同じ projectUrl が分かるなら同じ previewId で `submitEdit` をやり直す。分からなければ `previewEdit` からやり直す
- HTTP 404: previewId が使えない状態。再submitする前に `readPage` で意図した変更が既に反映されていないか確認する。未反映なら `previewEdit` からやり直す
- HTTP 409 `DuplicateTitle`: 新規作成の preview 後に同名ページが作られている。 `readPage` で同名ページを確認し、ユーザー意図が「既存に追記」なのか「別タイトルで新規」なのか判断する

### previewEdit/submitEdit 共通

- HTTP 401 / 403: 編集内容の復旧ではなく認証・権限の問題。 `login.md` を参照し、PAT・対象origin・project member権限を確認する。認証・権限を解決する前に同じ ops を繰り返し送らない
- HTTP 409 `NotFastForward`: preview生成後にページが更新されている。古い ops / previewId は再送せず、 `readPage` からやり直して `previewEdit` で新しい preview を作る
