# Cosenseのページを読む手順書

CLI コマンド一覧は [SKILL.md](SKILL.md) を参照する。
各コマンドの詳細仕様（引数・戻り値スキーマ・例）は、初めて使う前に必ず `cosense <command> --help` で確認する。

## 起点ページを決める

### URLが指定されている場合

「どこまで辿るかをユーザーに確認する」へ進む。

### 何らかのテーマや問いがある場合

#### 少なくとも5回ベクトル検索し、起点候補を選ぶ

`searchVector` を使う

##### ベクトル検索のコツ

ベクトル検索において、短いクエリは短いタイトルに、長いクエリは長いタイトルにマッチしやすい。

ユーザーの質問文をそのままベクトル検索に入力するのは効果的である。
長く具体的なタイトルのページには、そのトピックに関する具体的な情報が書かれている傾向がある。

ユーザーの質問文から単語を2〜3個抜き出して、まとめてベクトル検索するのも効果的である。
短く抽象的なタイトルのページには、重要な原理原則が書かれている事がある。

`searchVector` はOR検索できないので、複数キーワードを試したい時はクエリを分解して複数回叩く。

##### ベクトル検索結果の読み方

- `score` と `title` を総合判断して起点候補を複数選ぶ
- `exists` field に注目する。 起点として採用しない理由にはしない。 false のページも関連ページリストから文脈が掴めるため起点候補に含めてよい

#### 良い起点が見つからない場合

`searchVector` はタイトル＋本文中リンク記法だけが検索対象。 本文の語そのもので探したい時は `searchFullText` を使う。

1語で検索して0件でも「その語が無い＝情報が無い」と即断しない。送り仮名・カタカナの表記ゆれや類義語のバリアントも試す。

起点候補が決まったら「どこまで辿るかをユーザーに確認する」へ進む。

## どこまで辿るかをユーザーに確認する

起点ページが決まったら、最初の `browsePage` を呼ぶ前に AskUserQuestion で以下の2択を提示する。

| 選択肢                                  | 何が起こるか                                                                              | どんな時に向くか                 |
| --------------------------------------- | ----------------------------------------------------------------------------------------- | -------------------------------- |
| **関連ページまでしっかり読む** （推奨） | 起点ページを `browsePage` → 近傍メタデータを取得 → 必要なものを更に `browsePage` で深掘り | 概念の文脈や背景まで理解したい   |
| **指定されたページのみ読む**            | 起点ページを `browsePage` で1回だけ。他のページは辿らない                                 | 特定のページの内容だけ確認したい |

「指定されたページのみ読む」を選んでも、`browsePage` 出力末尾の関連タイトル一覧は読む対象に含まれる。これは `browsePage` 1回の出力に含まれており、追加のCLI呼び出しは発生しない。

### 質問をスキップする条件

以下の場合は AskUserQuestion を出さない。採用する方針は条件ごとに異なる。

#### ユーザーが既に深さを明示している場合

ユーザー指定に従う。

- 「このページだけ見て」「軽く読んで」「ざっと」等、浅さを示す指示 → 「指定されたページのみ読む」を採用
- 「関連ページも含めて」「しっかり調べて」「広く読んで」等、深さを示す指示 → 「関連ページまでしっかり読む」を採用

#### 別のSkillや上位Agentから呼び出され、ユーザーと直接対話していない場合

デフォルトとして「関連ページまでしっかり読む」を採用する。

### 方針はセッション全体に適用する

得られた方針はこの調査セッション全体に適用される。
起点候補が複数あっても質問は1回だけ。セッション中の追加 `browsePage` も AI が方針に沿って文脈判断する。ユーザーから明示的な方針変更がない限り再質問はしない。

## ページを読んでいく

### 起点ページを `browsePage` で読む

両モード共通の最初のステップ。

- 普段の閲覧はこれ1つで足りる。 URLに `#<lineId>` fragment が付いている時も `browsePage` が該当行を本文中でマークする
- `readPage` を併用するのは、 `previewEdit` の anchor 選定で全 `lines[].id` や `pageId` を取得したい時や、 本文を JSON として機械処理したい時だけ
- 出力末尾の関連ページのタイトルリストはぜひ眺めるべきだ。このページが属する概念の文脈が掴める
- 本文が空のページの場合は、特に関連ページリストを読むべきだ。意味は他ページの文脈の中で説明されている
- 起点候補が複数あれば、それぞれに対して `browsePage` する
- 人間と共同編集しているセッションでは、`browsePage` 出力の pageId（不変ID）と commitId を控えておく。後でページを見失った時に「変更を追う」で使う

### 「指定されたページのみ読む」を選んだ場合

起点ページ群を `browsePage` で読み終えたら終了する。他のページは辿らない。

`browsePage` 出力末尾の関連タイトル一覧は文脈の手がかりとして使ってよいが、未閲覧ページの本文内容は推測して断定しない。タイトルから読み取れる範囲のみを根拠にする。

本文が空のページが起点だった場合は特に、関連タイトル一覧しか情報がない。タイトルだけで答えられない問いには「本文に情報がない」「関連タイトルから推測すると…」と限界を明示して答える。

### 「関連ページまでしっかり読む」を選んだ場合

#### 近傍メタデータを取得する

近傍取得には以下のコマンドを状況に応じて使い分ける。

- `list1hopLinks`: 構造化JSONで1-hop近傍メタデータを一括取得する。`descriptions` / `relation` / `pageRank` を見て深掘り候補を選びたい時
- `search1hopLinks`: 既知のキーワードで1-hop近傍を絞り込みたい時
- `list2hopLinks`: 1-hopだけでは文脈が薄い時に、2-hop候補を構造化JSONで見る。全件読まず、 `title` / `descriptions` / `pageRank` で候補を絞る
- `search2hopLinks`: 2-hop範囲を既知のキーワードで絞る。広く探すより、仮説がある時に使う
- `browseRelatedPages`: 1-hop+2-hop のタイトル一覧を AI が読みやすい形式で眺める。2-hop 先まで広く文脈を掴みたい時
- `listPages`: プロジェクト全体から最近更新・被リンク数・タイトル順などで入口ページを探したい時
- `readProjectMembers`: 人名・アイコン・編集者の同定が回答に必要な時だけ使う

各要素の `relation` に注目する。参照方向は深掘り優先順位の判断材料になる。

#### 必要なものを深掘りする

- `descriptions` を眺めて、ユーザーの問いに関連するもの・注目度の高いものを `browsePage` で読む
- 全件読む必要はない。意図理解に必要な範囲で AI が判断する

## 変更を追う

共同編集中は、控えておいた pageId と commitId を `browsePageChanges` に渡すと、最後に読んだ後に共同編集者が何を変えたかを辿れる。pageId（ページの不変ID）と commitId（閲覧時点の最新コミット）は、`browsePage` 出力ではこの名前で、`readPage` の JSON では top-level の `id` / `commitId` として得られる。

### ページを見失ったら、リネームを疑う

さっきまで読めていたページが急に 404 や「(このページはまだ作成されていません)」になることがある。タイトルはURLになるため、共同編集者がタイトルを変更すると旧タイトルでは読めなくなるからだ。

このときは控えておいた pageId と commitId で `browsePageChanges <projectUrl> <pageId> --since <commitId>` を使う。タイトルが変わっても pageId は不変なので、リネームされても変更を辿れる。各コマンドの引数は `cosense browsePageChanges --help` で確認する。

### browsePageChanges 出力の読み方

- 冒頭の `title change` 行に注目する。変更後タイトルが示されていればリネームされている。新タイトルで `browsePage` を読み直す
- `## 変更` の各イベントは「誰がいつ何を変えたか」を時系列に示す。共同編集者が何を書いたかを把握し、自分の編集で上書き・重複させないようにする
- `--since` に前回の commitId を渡すと「前回読んだ後の変更」だけに絞れる

## 完了条件と回答

### 「指定されたページのみ読む」モード

起点ページ群を読み終えたら終了する。

### 「関連ページまでしっかり読む」モード

- そのページの意図・背景が理解できた状態で終了する。「全リンク読破」を完了条件としない
- 複数ページを調査した場合、必要な情報源が揃ったら終了する。集めても回答精度が上がらないと判断したら、見つかった範囲で答える

### 回答時

情報源（URL）を示しながらユーザーの問いに答える。
